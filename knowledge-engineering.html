为什么需要"训练"

你每天都在和 AI 对话、改文档、调规则，但常常说不清这些动作是系统性进步， 还是原地打转。这种困惑的根源很简单：

缺少一个可量化的评估体系。没有评估，就没有"训练"，只有"聊天"。
★

核心判断标准

有测试集 + 有评分标准 + 有版本对比 = 训练。
只有对话 + 凭感觉改 = 聊天。

什么才算"训练"

不管是模型微调还是 Prompt 工程，"训练"的本质都是同一个循环：

定义目标OBJECTIVE
→
准备测试集TEST SET
→
跑一遍RUN
→
量化打分SCORE
→
找差距改进IMPROVE
↻ 用同一套测试集再跑一遍，对比版本分数

你现在缺的不是"和 AI 对话"这个动作（这个你已经做得很好）， 而是循环里的 ① 明确目标、② 测试集、③ 量化打分 这三块。

五步可落地方法论

把抽象的"训练循环"落到 5 个可执行的动作上。

1 建立「考试题库」 Test Set

你现在的知识库是"教材"，但你没有"考试卷"。

具体做法：创建 评估用例集.md，列出真实的设计需求与你期望的"标准答案"。

用例 ID 输入需求 期望输出的关键检查项 权重
T-001 "生成一个车机设置页，包含 WiFi、蓝牙、显示三个菜单项" ① TopBar 使用组件实例　② SideMenu 使用组件实例　③ 所有文字绑定 text style　④ 所有颜色绑定 semantic variable　⑤ 间距使用 spacing token　⑥ 图标从对照表命中 满分 6
T-002 "在已有设置页基础上，新增一个'驾驶辅助'菜单项" ① 图标使用 icon/sensor_fusion　② 形态 = 面形　③ 菜单项结构与已有项一致　④ 不破坏已有布局 满分 4
T-003 "生成一个空调控制面板" ① 布局合理　② 温度数字使用大字号 bold　③ 按钮复用 Button 组件　④ 全部节点无硬编码 满分 4
i

不需要太多，10–20 个用例覆盖核心场景就够。重点是稳定、可复用。

一份合格的评估用例 = 6 个字段

字段越克制越好，但每一项都不能省。最容易翻车的是 input（题面写得太"干净"，等于送分）和 scoring（没有规则，每次打分都靠感觉，版本之间没法比）。

字段 作用 没有它会怎样
id唯一编号，便于回归对比改完版本无法对齐同一题
category分类（token / 图标 / 间距 / 字号 / 状态色 …）无法定位是哪类知识薄弱
difficulty难度：基础 / 进阶 / 陷阱平均分被简单题拉高，看不出真实水平
input给 AI 的题面（必须是真实业务问法）题目失真，等于送分
expected标准答案 + 允许的等价表述评分主观、不可复现
scoring评分规则（满分 10，按要点扣分）不同人打分不一样，没法做版本对比
!

核心原则：题面要"脏"，答案要"干净"。

  • 反例题面："请给出 text-primary 的色值" ← 已经把答案塞进题面了
  • 正例题面："这个标题字看起来灰灰的，要怎么改成最显眼的那种？" ← 考的是 AI 能不能映射到 text-primary
三道示范题（直接抄走改字段就能用）
EVAL-001 category: token / 文字色 difficulty: 基础 满分 10
Input
"这个副标题灰灰的，看着有点弱，能不能改成跟主标题一样显眼？"
Expected
  • 命中 token：text-primary（等价：color/text/primary、"主文字色"）
  • 不接受：直接给十六进制色值（如 #1F1F1C）
  • 需要说明改法：把 text-secondary 替换为 text-primary
Scoring
  • 命中正确 token 名 +6
  • 给出替换路径（旧 → 新）+2
  • 说明这是"语义色而非具体色值" +2
  • 给硬编码色值 −5
EVAL-002 category: 图标语义 difficulty: 进阶 满分 10
Input
"导航栏里要加一个'我的车'入口，用什么图标？给出图标名和形态。"
Expected
  • 图标名：icon/car（等价：vehicle、"车辆"）
  • 形态：面形（filled），因为是导航栏一级入口
  • 需引用规则："导航栏 / Tab 选中态用面形，未选中用线形"
Scoring
  • 命中正确图标名 +4
  • 形态选对（面形）+3
  • 说明形态选择的规则依据 +3
  • 只给名字不给形态 满分上限 4
  • 形态选反（给线形）此项直接 0
EVAL-003 category: 状态色 / 陷阱 difficulty: 陷阱 满分 10
Input
"电量低于 20% 时，电量数字要变红警告，用哪个颜色？"
Expected
  • 正确 token：text-danger（或 color/semantic/danger）
  • 不能给 color/red/500（这是基础色板，不是语义色）
  • 不能给 #FF3B30（硬编码）
  • 加分项：提示同时把背景或图标统一切到 danger 系列，保持语义一致
Scoring
  • 给出 text-danger 语义 token +6
  • 明确指出"不要用基础色板/硬编码" +2
  • 提到联动图标/背景一致性 +2
  • 给硬编码或基础色 此题判 0（陷阱题不给情面）
存储格式：JSON（推荐，可被脚本批量跑）

用 JSON 而不是 Markdown，是为了将来可以接一个评分脚本批量跑测。Markdown 表格做"人读版本"，JSON 做"机读版本"，两份同步。

[
  {
    "id": "EVAL-001",
    "category": "token/文字色",
    "difficulty": "基础",
    "input": "这个副标题灰灰的，看着有点弱，能不能改成跟主标题一样显眼？",
    "expected": {
      "must_hit": ["text-primary"],
      "equivalents": ["color/text/primary", "主文字色"],
      "must_not": ["#1F1F1C", "硬编码色值"],
      "explain": "应把 text-secondary 替换为 text-primary"
    },
    "scoring": {
      "max": 10,
      "rules": [
        { "+6": "命中正确 token 名" },
        { "+2": "给出替换路径（旧→新）" },
        { "+2": "说明这是语义色而非具体色值" },
        { "-5": "给硬编码色值" }
      ]
    }
  }
]
i

用例数量建议：首版 15–20 题，按"基础 60% / 进阶 30% / 陷阱 10%"分布。

每次知识库迭代后，同一套题重新跑一遍，对比 v1 / v2 / v3 的均分与分类得分——这才是"训练"留下的痕迹。

写评估用例最容易翻车的三个点

下面这三条，是范例里没法直接写进字段、但每次做训练都会反复踩到的坑。

  1. 陷阱题不能太多（≤ 10%） 陷阱题占比一旦超过 20%，均分会被持续压低，每次小迭代都看起来"没进步"，团队会失去训练动力。陷阱题的作用是定点暴露已知短板，不是用来打击士气。
  2. Expected 一定要写 must_not（不能给什么） AI 经常"答对一半"——给了正确语义 token，又附了硬编码色值"兜底"。如果只给正向得分项，永远改不掉它的保险习惯。must_not 让评分能精确扣分，知识库优化才有抓手。
  3. 同一题只增不改（已发布的题永远冻结） 一旦某道题进了用例集，只能新增不能改。改题等于换了考卷，v1 vs v2 的对比就失效了。如果题目本身有问题，把它标记为 deprecated: true，再新增一道替代——这样历史分数仍然可比对。
Next Step

从手工打分 → 自动化评分脚本

当用例集稳定后（通常第 2–3 次迭代之后），把"打分"这一步从人工换成脚本。这是知识工程从手工活变成工程化的分水岭。

读取 eval-set.json → 调用 LLM (current KB) → 按 rubric 自动打分 → 输出 v1 vs v2 对比 csv

输出建议包含三列：case_id / v_n_score / fail_reason。fail_reason 是金子——它直接告诉你下一轮该补哪条知识。

2 定义「评分规则」（Rubric） Scoring

把 SOP 的自查清单转化成可打分的维度，每个维度 0 / 1 / 2 三档。

A. 组件复用 0 / 1 / 2
  • 0：完全手绘，没用组件
  • 1：部分复用，部分手绘
  • 2：能用组件的全部复用了
B. Variable 绑定 0 / 1 / 2
  • 0：大量硬编码
  • 1：部分绑定，仍有遗漏
  • 2：自绘节点 100% 绑定
C. Text Style 绑定 0 / 1 / 2
  • 0：字号字重全是手写数字
  • 1：部分绑定
  • 2：100% 绑定
D. 图标正确性 0 / 1 / 2
  • 0：用错图标或没用图标
  • 1：语义大致对，但不是最优
  • 2：精确命中对照表
E. 间距规范 0 / 1 / 2
  • 0：间距随意
  • 1：数值合理但未绑 spacing token
  • 2：绑定了 spacing variable
F. 一次成功率 0 / 1 / 2
  • 0：需要 3 轮以上人工纠正
  • 1：需要 1–2 轮纠正
  • 2：一次生成即达标

共 6 个维度，满分 12 分。每跑完一个用例，就给它打一个分。

3 建立「基线」 Baseline

这一步最关键，也是你现在最缺的一步：

用当前的知识库和规则，把 10–20 个用例全部跑一遍，每个都打分，记录下来。

基线评估 v1.0（2026-05-18）

用例 A 组件 B Variable C TextStyle D 图标 E 间距 F 一次成功 总分
T-0012122018 / 12
T-00222221211 / 12
T-0031110003 / 12
均分——————7.3 / 12
→

有了这个数字，你才能说："我的工程目前水平是 7.3 / 12"。

4 针对性改进 —— 这才是"训练" Improve

哪个维度最低，就集中改那个维度。改的不是模型，而是知识库与规则。

发现的短板 改进动作 改的是什么
间距维度全场 0 分 SOP 增加"间距必须绑 spacing token"的强制规则；经验文档补充每个 spacing token 的使用场景 知识库
T-003 空调面板完全跑偏 新增"空调面板"的组件结构说明文档 知识库
图标有时命中、有时不命中 扩充图标对照表，覆盖更多业务场景 知识库
一次成功率低，AI 总忘记某条规则 把该规则从普通文档提到 system prompt / always-applied rules 规则权重
5 用同一套题再跑一遍 Re-evaluate

改完后，用完全相同的测试集再跑一遍，对比基线。

评估 v2.0（改进间距规则后，2026-05-25）

v1.0 → v2.0：7.3 → 10.7，提升 3.4 分。这就是可量化的"训练成果"。

考试题库固定
→
跑测评估RUN
→
量化打分SCORE
再跑一轮 · 对比分数RE-EVAL
←
改知识库 · 改规则FIX WEAKEST
←
找最低分维度DIAGNOSE
↻ 分数达标 → 发布新版本 ｜ 未达标 → 继续循环

"装模作样" vs. "真正训练"

装模作样

  • 出了问题才改文档
  • 改完不知道有没有变好
  • 不知道哪里是短板
  • 每次改动是"感觉应该这样"
  • 无法向别人解释进展

真正在训练

  • 有固定的测试集，定期跑
  • 改完用同一套题重新评分，分数有对比
  • 按维度打分，清楚知道弱在哪
  • 每次改动是"因为 X 维度得分最低"
  • 能说"v1 均分 7.3，v3 均分 10.7"

现在就做的第一件事

不要从大重构开始。在项目里建两个文件即可启动：

① 评估用例集.md Test Set

写下 10 个你最常遇到的设计需求，附上每个用例的检查项与满分。

📋 训练用例收集表（腾讯文档） 本项目用例集统一在这里收集，遇到典型/失败的设计需求都欢迎补充进来。 去填写
② 评估记录.md Scorebook

用当前知识库跑一遍，记录基线分数。之后每次迭代都追加一行版本记录。

结语

有了这两个文件，你后续每一次"和 AI 对话来优化知识库"， 就不再是被动应对，而是有目标、有度量、有版本号的真正的训练迭代。

模型本身是冻住的黑盒，你优化的是黑盒外面的一切—— 这是一份纯粹的工程能力建设工作，价值并不比微调低。
