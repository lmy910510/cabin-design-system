为什么需要"训练"

你每天都在和 AI 对话、改文档、调规则，但常常说不清这些动作是系统性进步， 还是原地打转。这种困惑的根源很简单：

缺少一个可量化的评估体系。没有评估，就没有"训练"，只有"聊天"。
★

核心判断标准

有测试集 + 有评分标准 + 有版本对比 = 训练。
只有对话 + 凭感觉改 = 聊天。

什么才算"训练"

不管是模型微调还是 Prompt 工程，"训练"的本质都是同一个循环：

定义目标OBJECTIVE
→
准备测试集TEST SET
→
跑一遍RUN
→
量化打分SCORE
→
找差距改进IMPROVE
↻ 用同一套测试集再跑一遍，对比版本分数

你现在缺的不是"和 AI 对话"这个动作（这个你已经做得很好）， 而是循环里的 ① 明确目标、② 测试集、③ 量化打分 这三块。

五步可落地方法论

把抽象的"训练循环"落到 5 个可执行的动作上。

1 建立「考试题库」 Test Set

你现在的知识库是"教材"，但你没有"考试卷"。

具体做法：创建 评估用例集.md，列出真实的设计需求与你期望的"标准答案"。

📋 训练用例收集表（腾讯文档） 本项目的"考试题库"统一在这里收集，遇到典型/失败的设计需求都欢迎补充进来。 去填写

1.1　题库怎么组织

不要给题贴 L1/L2/L3/L4 之类的能力标签——所有题统一打分，标签反而是负担。题库就是一个池子，按 Agent 归属 和 颗粒度 两个简单维度组织即可。

维度 A：Agent 归属

区分"谁负责出题、谁负责答题"。

  • 公共卷：纯设计系统题（token / 组件 / 通用图标），三个 Agent 共用
  • Agent 1 专业卷：长途出行场景（导航、续航、路况……）
  • Agent 2 专业卷：POI 场景（详情、推荐、商户态……）
  • Agent 3 专业卷：点单场景（菜单、购物车、下单流程……）
维度 B：题目颗粒度

从一句话需求到跨页面流程，颗粒度越大越接近真实业务，但越难给"标准答案"。题库要金字塔形——原子题最多（便宜可复用），流程题最少（贵但最像真实考试）。

  • 原子题：一句话需求（"一个开关项"）
  • 组件题：一个完整组件（"餐饮卡片"）
  • 页面题：一个完整页面（"POI 详情页"）
  • 流程题：跨页面或跨状态（"下单全流程 3 屏"）
推荐配比（共 ~24 题打底）

不是每个格子都要填满。下面这个配比的逻辑：公共卷把规范底座铺厚，每个 Agent 各出 4–5 道贴自己业务的题，再留少量陷阱题（场景对抗）当兜底测试。

原子题 组件题 页面题 流程题
公共卷三个 Agent 共用 6token / 图标 / 排版 3TopBar / 卡片 / 列表 1设置页 —
Agent 1 专业卷长途出行 1陷阱题 1提示卡片 1路线规划页 1续航对抗
Agent 2 专业卷POI — 1POI 卡片 1详情页 1闭店流程
Agent 3 专业卷点单 — 1菜单项 1购物车页 2下单 + 打烊对抗
i

公共卷 + 专业卷：公共卷考的是设计系统本身（组件、token、图标），三个 Agent 共用一份；专业卷考的是 Agent 对自己场景的理解，必须各出一份。以后再加第 4 个 Agent，只需要补它的专业卷，公共卷不用重写。

少量"陷阱题"（题面藏冲突或边界条件，比如"续航 80 公里去 200 公里外"）混在 Agent 专业卷里就够了，不用单独划一个 L4 类别——打分时它跟普通题一样走五维度。

1.2　一道用例长什么样（6 个字段）

字段越克制越好，但每一项都不能省。最容易翻车的是 input（题面写得太"干净"，等于送分）和 scoring（没有规则，每次打分都靠感觉，版本之间没法比）。

字段 作用 没有它会怎样
id唯一编号，便于回归对比改完版本无法对齐同一题
category定位：公共卷 / Agent 专业卷 × 原子/组件/页面/流程无法定位题该归谁出、谁答
difficulty难度：基础 / 进阶 / 陷阱平均分被简单题拉高，看不出真实水平
input给 AI 的题面（必须是真实业务问法）题目失真，等于送分
expected标准答案 + 允许的等价表述（含 must_not）评分主观、不可复现
scoring评分规则，统一五维度（每维 0–2，满分 10）没有规则每次打分都靠感觉，版本之间没法比
!

核心原则：题面要"脏"，答案要"干净"。

  • 反例题面："请给出 text-primary 的色值" ← 已经把答案塞进题面了
  • 正例题面："这个标题字看起来灰灰的，要怎么改成最显眼的那种？" ← 考的是 AI 能不能映射到 text-primary

1.3　范例库：10 道题铺满全部考点

三条最容易翻车的经验

  • 陷阱题 ≤ 10%：超过会持续压低均分，团队失去训练动力。
  • Expected 必须写 must_not：AI 经常"答对一半"还附带硬编码兜底，没有反向项就改不掉这个保险习惯。
  • 已发布的题只增不改：改题等于换考卷，v1 vs v2 对比就失效了。要废弃就标记 deprecated: true 再新增。
题目要全部手写吗？——手写标杆 + AI 扩变体 思辨补充

分工：公共卷（规范类）标杆题手写后 AI 扩变体；Agent 专业卷必须人写。AI 不知道你的"打烊要前置""续航不足主动报警"这种业务红线，编出来全是通用陷阱，没鉴别力。

体量参考（3 个 agent）：手写约 12–20 道，AI 扩到 24–40 道，一两个下午能搞定第一轮。

AI 扩题提示词（直接复制）

你是测试用例设计师。基于下面这道标杆题，出 N 道变体。

【标杆题】粘贴上面 10 道里的任意一道

【硬约束】
1. 题的归属（公共卷 / 哪个 Agent 专业卷）和颗粒度不许变
2. 换组件 / 换场景，但考点同源
3. 题面必须是"用户口语"，不能把答案 token 写进题面
4. 每题给出：Input / Expected（含 must_not）/ Scoring（按五维度：组件 / 样式 / 信息层级 / 场景适配 / 信息密度）
5. 公共卷题禁止混入 Agent 专属业务知识；Agent 专业卷题禁止只考通用规范

输出 N 道，编号 V1–VN。
Next Step

从手工打分 → 自动化评分脚本

当用例集稳定后（通常第 2–3 次迭代之后），把"打分"这一步从人工换成脚本。这是知识工程从手工活变成工程化的分水岭。

读取 eval-set.json → 调用 LLM (current KB) → 按 rubric 自动打分 → 输出 v1 vs v2 对比 csv

输出建议包含三列：case_id / v_n_score / fail_reason。fail_reason 是金子——它直接告诉你下一轮该补哪条知识。

2 定义「评分规则」（Rubric） Scoring

核心一句话：每道题都用同一套五个维度打分，每维 0–2，满分 10。不再分"规范分 / 业务分"两栏，也不再按 L1–L4 拆——所有题一把尺子，打完五个维度就出分。

★

五个维度由"不犯错"逐级递进到"会做设计"

  1. 组件　——　用没用对（关键组件有没有自绘）
  2. 样式　——　token 绑没绑（颜色 / 字号 / 间距 / 圆角）
  3. 信息层级　——　主次出没出来（一眼能不能看出谁主谁次）
  4. 场景适配　——　这设计放进真实场景能不能用（车机视距 / 驾驶态 / 光照）
  5. 信息密度　——　该有的有、不该有的没有（精炼 vs 一团乱）

前 2 条 = 不犯错（规范底线）／第 3 条 = 设计基本功／后 2 条 = 真正的设计能力。

五维度打分细则

维度 0 分 1 分 2 分
1. 组件 关键组件自绘（自己画了 Card / Button / Tab） 主要组件用了组件库，但有少量自绘或属性改坏了 所有组件都来自组件库，属性没乱改
2. 样式 大量自定义颜色 / 字号 / 间距，没走 token 主要走了 token，但个别地方写死值 颜色 / 字体 / 间距 / 圆角全部走 Variable / Text Style
3. 信息层级 所有元素一样大一样色，看不出主次 大体能看出主次，但个别地方层级模糊 主操作 / 主信息一眼跳出，次要信息明显退到次级
4. 场景适配 违反场景约束（驾驶时字号太小、夜间用纯白底） 基本可用，但有少数地方没考虑场景 字号 / 对比度 / 触控目标 / 操作密度都贴合场景
5. 信息密度 关键信息漏了，或塞了一堆无关信息 关键信息都在，但有冗余 / 缺辅助说明 信息刚好够用——题面要求全到位、没有视觉噪声

用一道真题走一遍（A1-COMP-001 拥堵绕行）

同一道题，差稿 vs 好稿，用五维度打分能拉出明显差距：

维度 差稿 好稿 为什么差距在这
1. 组件 1 2 差稿自绘了 Card 背景；好稿全用组件实例
2. 样式 1 2 差稿写了 #FF3B30；好稿绑 color/danger
3. 信息层级 0 2 差稿三个按钮一样大；好稿"绕行"是 Primary、"12 分钟"用强调字号
4. 场景适配 0 2 差稿正文 14px 驾驶时根本看不清；好稿 24px + 高对比
5. 信息密度 1 2 差稿塞了 4 行说明文字；好稿只留"绕行 / 不绕行 / 多花 12 分钟"三件事
合计 3 / 10 10 / 10 同一题不同稿能拉开 7 分差距，信号足够强
i

关于"N/A"：纯静态展示题（比如"次要文字用什么色"）天然没有"场景适配"或"信息密度"可考——这些维度标 N/A，最后按实际打分维度数按比例换算到满分 10。比如只考样式 + 层级两维（满 4），实际得 3，则总分 = 3 ÷ 4 × 10 = 7.5。

关于"陷阱红线"：陷阱题（题面藏冲突或边界条件）允许定一条"答错此条整题判 0"的硬规则——例如"打烊提示前置"、"续航不足主动报警"。红线只用在陷阱题里，普通题不要乱设。

关于这套打分的诚实边界 思辨补充

这套打分依然是半黑盒。"信息层级 0/1/2"还是靠人判断，不可能完全客观。但比起"规范分 / 业务分"那种抽象分类，这五维度问题更聚焦：

  • "关键组件有没有自绘" → 看图层就知道（白盒）
  • "颜色 / 间距是不是 token" → 扫一遍就知道（白盒）
  • "主次能不能一眼看出" → 比"业务做得怎么样"具体多了（半白盒）
  • "字号在驾驶视距下够不够" → 有物理判据（半白盒）

务实做法：同一个版本的所有题让同一个人打分，消除评估人偏差；版本对比时关注五维度均值的趋势，而不是绝对值。等跑过 2–3 版稳定下来，再把争议最大的几道题升级成更细的 checklist。

!

不再使用以下术语：L1 / L2 / L3 / L4 能力线、规范分、业务分、能力维度。统一只用：组件 / 样式 / 信息层级 / 场景适配 / 信息密度 这五个词。术语一致是版本对比能成立的前提。

3 建立「基线」并跑出对比 Baseline & Re-eval

用当前的知识库把全部用例跑一遍，逐条按五维度打分，得到 v1 基线；改完后用完全相同的题再跑一遍，得到 v2。

v1 → v2 评估对比示例（按维度看均值）

不要看每道题的总分变化——粒度太细信号弱。直接看五个维度在所有题上的均值，哪个维度均值低、哪个维度涨得多，下一步该补什么一目了然。

维度 v1 均值 v2 均值 Δ 解读
1. 组件 1.4 1.9 +0.5 知识库强化了"必须用组件库"后基本封顶
2. 样式 1.6 1.9 +0.3 token 绑定也接近封顶，规范层稳定
3. 信息层级 0.8 1.5 +0.7 这版的真正进步：主次按钮、字号层级有意识做了
4. 场景适配 0.6 0.9 +0.3 仍是短板：驾驶视距、夜间对比度还经常忽略
5. 信息密度 1.0 1.2 +0.2 变化不大，AI 还是倾向"什么都给"
总均值（×5） 5.4 7.4 +2.0 —
↑

看维度均值，不看总分。"信息层级 0.8 → 1.5 是这版主战场，场景适配 0.6 → 0.9 仍是短板，下一版重点补车机视距 / 驾驶态适配的知识。"——这种话才是可量化、可解释的训练成果。

4 针对性改进 —— 这才是"训练" Improve

哪个维度均值最低，就集中改哪个维度。改的不是模型，而是知识库与规则。

短板维度 典型表现 改进动作 改的是什么
1. 组件 AI 经常自绘卡片背景、自画按钮 SOP 加"任何卡片 / 按钮交付前必须是组件实例"，并在公共卷加"自绘 = 0 分"硬约束 设计系统主知识库
2. 样式 间距 / 颜色偶尔写硬编码值 SOP 增加"自绘节点交付前 100% 绑 token"的强制自查项 设计系统主知识库
3. 信息层级 三个 agent 都把决策动作做成了次按钮 在每个 agent 的业务规范里写死："唯一决策动作必须是主按钮，且不可折叠" Agent 业务知识库
4. 场景适配 POI 闭店、餐饮打烊等异常态 AI 经常忘做；车机视距字号偏小 每个 agent 文档加"必须覆盖的异常态清单"+ 公共卷加"车机最小字号 / 触控目标"硬指标，做成 always-applied rule Agent 业务知识库 + 规则权重
5. 信息密度 AI 倾向"什么都给"，主信息被次要信息淹没 公共卷加"一屏一个主 CTA"原则；每个 agent 给出"该屏必须出现 / 必须不出现"清单 Agent 业务知识库

改完后回到第 3 步，用同一套题重新跑分——这就是闭环。

"装模作样" vs. "真正训练"

装模作样

  • 没有固定测试集，凭感觉改
  • 改完不知道有没有变好
  • 说不清当前水平、说不清下一步

真正在训练

  • 有固定题库，按 公共卷 / Agent 专业卷 × 颗粒度 组织
  • 每题统一五维度打分（组件 / 样式 / 信息层级 / 场景适配 / 信息密度）
  • 能说出"信息层级 0.8 → 1.5、场景适配仍是短板"

结语

从今天起，把"和 AI 对话来优化知识库"变成有目标、有度量、有版本号的迭代—— 题库 + 评分 + 版本对比，三件事就足够撑起这套循环。

模型本身是冻住的黑盒，你优化的是黑盒外面的一切——这是一份纯粹的工程能力建设工作，价值并不比微调低。
