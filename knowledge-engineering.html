为什么需要"训练"

你每天都在和 AI 对话、改文档、调规则，但常常说不清这些动作是系统性进步， 还是原地打转。这种困惑的根源很简单：

缺少一个可量化的评估体系。没有评估，就没有"训练"，只有"聊天"。
★

核心判断标准

有测试集 + 有评分标准 + 有版本对比 = 训练。
只有对话 + 凭感觉改 = 聊天。

什么才算"训练"

不管是模型微调还是 Prompt 工程，"训练"的本质都是同一个循环：

定义目标OBJECTIVE
→
准备测试集TEST SET
→
跑一遍RUN
→
量化打分SCORE
→
找差距改进IMPROVE
↻ 用同一套测试集再跑一遍，对比版本分数

你现在缺的不是"和 AI 对话"这个动作（这个你已经做得很好）， 而是循环里的 ① 明确目标、② 测试集、③ 量化打分 这三块。

五步可落地方法论

把抽象的"训练循环"落到 5 个可执行的动作上。

1 建立「考试题库」 Test Set

你现在的知识库是"教材"，但你没有"考试卷"。

具体做法：创建 评估用例集.md，列出真实的设计需求与你期望的"标准答案"。

📋 训练用例收集表（腾讯文档） 本项目的"考试题库"统一在这里收集，遇到典型/失败的设计需求都欢迎补充进来。 去填写

1.1　题库怎么分类与组织

很多人卡壳是因为脑子里只用一个维度排所有题（基础 / 进阶），但真实题库需要两个独立维度交叉。

维度 A：考什么能力

基础规范题做满分 ≠ 业务理解过关。两条能力线必须各自打分，否则版本对比时你说不清是哪条进步了。

  • L1 基础规范：token / 组件 / 排版 / 图标使用是否合规
  • L2 业务理解：是否理解 agent 的目标用户、场景、信息优先级
  • L3 跨规范泛化：多 agent 共用规范时能否保持一致
  • L4 边界 / 容错：给残缺、冲突或陷阱输入时的表现
维度 B：题目颗粒度

从一句话需求到跨页面流程，颗粒度越大越接近真实业务，但越难给"标准答案"。题库要金字塔形——原子题最多（便宜可复用），流程题最少（贵但最像真实考试）。

  • 原子题：一句话需求（"一个开关项"）
  • 组件题：一个完整组件（"餐饮卡片"）
  • 页面题：一个完整页面（"POI 详情页"）
  • 流程题：跨页面或跨状态（"下单全流程 3 屏"）
A × B 矩阵 = 题库骨架（推荐配比）

不是每个格子都要填满。下面是一个共 ~24 题的打底配比，重点放在规范题密集、业务题够重、每层都种一个陷阱。

原子题 组件题 页面题 流程题
L1 基础规范规范本身的考试 6token / 图标 / 排版 3TopBar / 卡片 / 列表 1设置页 —
L2 业务理解每个 agent 必须有 — 3每 agent 1 道 3每 agent 1 道 2核心流程
L3 跨规范泛化三个 agent 共用一致性卷 — 2共用按钮 / 图标语义 1一致性巡检 —
L4 边界 / 容错每层种一个陷阱 1 1 1 1
i

公共卷 + 专业卷：L1 和 L3 是"设计系统本身"的考试，三个 agent 共用一份；L2 和 L4 是每个 agent 的"专业卷"，必须各出一份。以后再加第 4 个 agent，只需要补它的专业卷，公共卷不用重写。

为什么是 2 维，不是 3 维或 5 维？ 思辨补充

2 维不是最优解，是当前规模下的甜区。理论上可对到 Bloom 教育目标分类法的二维矩阵 + 软件测试金字塔；机器学习视角上，两个维度尽量正交是为了避免"分数涨了但说不清涨在哪"的因子纠缠。

不推荐一上来就 3+ 维的现实原因：

  • 组合稀疏：~24 道题铺到 4×4×4=64 格，每格不到一道，方差盖过信号。
  • 评分成本：维度每加一条，打分工时是乘法增长，不是加法。
  • 决策粒度：2 维已能回答"补规范库 / 补业务库 / 补一致性 / 补容错"四类决策；再细团队反应不过来。

什么时候该升级到 3 维：Agent > 5 时加 Agent 维度；题库 > 100 时加 难度维度；做多模型对比时加 模型维度。判据：当某个改进决策在现有维度里找不到对应格子，那个方向就是该加的下一维。

1.2　一道用例长什么样（6 个字段）

字段越克制越好，但每一项都不能省。最容易翻车的是 input（题面写得太"干净"，等于送分）和 scoring（没有规则，每次打分都靠感觉，版本之间没法比）。

字段 作用 没有它会怎样
id唯一编号，便于回归对比改完版本无法对齐同一题
category定位：L1–L4 × 原子/组件/页面/流程无法定位是哪条能力线薄弱
difficulty难度：基础 / 进阶 / 陷阱平均分被简单题拉高，看不出真实水平
input给 AI 的题面（必须是真实业务问法）题目失真，等于送分
expected标准答案 + 允许的等价表述（含 must_not）评分主观、不可复现
scoring评分规则，规范分 + 业务分两栏分不开是规范涨了还是业务理解涨了
!

核心原则：题面要"脏"，答案要"干净"。

  • 反例题面："请给出 text-primary 的色值" ← 已经把答案塞进题面了
  • 正例题面："这个标题字看起来灰灰的，要怎么改成最显眼的那种？" ← 考的是 AI 能不能映射到 text-primary

1.3　范例库：10 道题铺满 L1–L4

下面 10 道题覆盖 4 条能力线 × 4 种颗粒度 × 3 个 agent，是一份可以直接照抄结构的最小完整样本。看一遍就能明白"题该长什么样"，剩下的照葫芦画瓢。

L1-001 L1 规范 · 原子 公共卷
Input
"这个次要说明文字看起来有点抢，想让它退到背景里一点，用哪个色？"
Expected
  • 正确：text-secondary 或 text-tertiary 语义 token
  • must_not：color/gray/500、#999999、opacity: 0.6
Scoring
规范 6 + 说明 secondary/tertiary 用法差异 +2 ＝ 满分 8
L1-002 L1 规范 · 组件 公共卷
Input
"做一个'确认删除'的弹窗，要有取消和确认两个按钮"
Expected
  • 复用 Dialog 组件实例 + Button 组件实例；主按钮 variant=danger，次按钮 variant=ghost
  • must_not：自绘黑底浮层、两个按钮都 primary、确认按钮放左边
Scoring
组件复用 +4；按钮主次正确 +3；危险动作用 danger 变体 +3 ＝ 满分 10
L1-003 L1 规范 · 页面 公共卷
Input
"生成一个车机设置页，包含 WiFi、蓝牙、显示三个菜单项"
Expected
① TopBar 复用组件实例　② 列表项复用组件实例　③ 文字绑定 text style　④ 颜色绑定 semantic variable　⑤ 间距走 spacing token　⑥ 图标命中对照表
Scoring
每命中一项 +1，满分 6；任一项写死硬编码 直接扣 2
L2-A1-001 L2 业务 · 组件 Agent 1 · 长途出行
Input
"前方拥堵，建议绕行，多花 12 分钟"——做一个驾驶中的提示卡片
Expected
  • 规范分：复用 Card / Button 组件，文字 / 颜色 / 间距全部 token 化
  • 业务分：① 决策动作（绕 / 不绕）必须是主按钮　② "12 分钟"作为代价要醒目（次级强调而非次要文字）　③ 不能折叠到二级菜单
  • must_not：把"绕行"做成纯文字链；把"12 分钟"用 text-tertiary 显示
Scoring
规范 4 + 业务 6 ＝ 满分 10
L2-A2-001 L2 业务 · 页面 Agent 2 · POI
Input
"POI 详情页：一家咖啡馆，要有简介 / 评分 / 玩法 / 周边"
Expected
  • 规范分：Hero 图、Tag、Button 全部走组件；零硬编码色值
  • 业务分：① 核心动作（导航 / 收藏 / 分享）一屏可达　② 评价 / 玩法 / 周边分区清晰　③ 广告位不能压住主信息
  • must_not：把"导航"按钮藏在二级 Tab、用满屏 banner 推销
Scoring
规范 4 + 业务 6 ＝ 满分 10
L2-A3-001 L2 业务 · 流程 Agent 3 · 点单
Input
"选菜 → 购物车 → 下单 → 等待"4 屏完整点单流程
Expected
  • 规范分：4 屏组件复用一致，无自绘卡片或硬编码尺寸
  • 业务分：① 每屏只有一个主 CTA　② 等待页有进度反馈　③ 异常态（缺货 / 支付失败）预留位置
  • must_not：让用户在购物车页和下单页都看到两个并列的"确认"按钮
Scoring
规范 4 + 业务 6 ＝ 满分 10
L3-001 L3 跨域一致性 · 页面 公共卷
Input
"我在 POI 详情页用了 Button-primary 做'导航'按钮，在路线规划页也有'开始导航'按钮——两个页面要保持一致吗？怎么保证？"
Expected
  • 必须同 variant（primary）+ 同尺寸档位 + 同图标（命中对照表里"导航"那条）
  • 判断标准：用户在两个页面看到这个按钮时，能否一眼认出"这是同一个操作"
  • must_not：一个用 primary 一个用 ghost、一个带图标一个不带、一个写"导航"一个写"开始导航"
Scoring
指出 variant/尺寸/图标三要素 +6；说明文案也要统一 +2；提到通过共享 token 而非各自绑定来保证 +2 ＝ 满分 10
L4-TRAP-001 L4 陷阱 · 原子 公共卷
Input
"电量低于 20% 时，电量数字要变红警告，用哪个颜色？"
Expected
  • 正确：text-danger（或 color/semantic/danger）
  • must_not：color/red/500（基础色板，不是语义色）；#FF3B30（硬编码）
Scoring
语义 token +6；明确反对基础色板/硬编码 +2；提到联动图标/背景一致性 +2；给出硬编码或基础色 此题判 0（陷阱题不给情面）
L4-TRAP-A1-001 L4 陷阱 · 流程 Agent 1 · 长途出行
Input
"用户输入：导航到 200 公里外的目的地，当前续航 80 公里"
Expected
  • 规划路线前必须主动报警"续航不足"
  • 同步给出沿途充电站建议（至少 1 个）
  • 主按钮应该是"加入充电站"而非"继续导航"
  • must_not：直接出路线 + 一行小字"注意电量"；把充电站藏在二级菜单
Scoring
主动报警 +4；给出充电站 +3；主按钮指向充电而非继续 +3；任何一条缺失此题判 0
L4-TRAP-A3-001 L4 陷阱 · 流程 Agent 3 · 点单
Input
"用户在 21:55 浏览菜单，门店 22:00 打烊"
Expected
  • 菜单页顶部必须有"5 分钟后打烊"醒目提示（warning 级别，非次要文字）
  • 加入购物车按钮可点，但点结算前需要二次确认
  • must_not：让用户走完整个下单流程到支付完成才告知打烊；用 text-tertiary 显示提示
Scoring
提示前置 +4；视觉权重达 warning 级 +3；结算前二次确认 +3；下单后才告知 此题判 0
i

三条最容易翻车的经验

  • 陷阱题 ≤ 10%：超过会持续压低均分，团队失去训练动力。
  • Expected 必须写 must_not：AI 经常"答对一半"还附带硬编码兜底，没有反向项就改不掉这个保险习惯。
  • 已发布的题只增不改：改题等于换考卷，v1 vs v2 对比就失效了。要废弃就标记 deprecated: true 再新增。
题目要全部手写吗？——手写标杆 + AI 扩变体 思辨补充

分工：L1/L3 标杆题手写后 AI 扩变体；L2/L4 业务题必须人写。AI 不知道你的"打烊要前置"、"续航不足主动报警"这种业务红线，编出来全是通用陷阱，没鉴别力。

体量参考（3 个 agent）：手写约 12–20 道，AI 扩到 24–40 道，一两个下午能搞定第一轮。

AI 扩题提示词（直接复制）

你是测试用例设计师。基于下面这道标杆题，出 N 道变体。

【标杆题】粘贴上面 10 道里的任意一道

【硬约束】
1. 能力线和颗粒度不许变
2. 换组件 / 换场景，但考点同源
3. 题面必须是"用户口语"，不能把答案 token 写进题面
4. 每题给出：Input / Expected（含 must_not）/ Scoring（≤3 条）
5. L1 题禁止混入业务知识，L2 题禁止混入纯规范题

输出 N 道，编号 V1–VN。
Next Step

从手工打分 → 自动化评分脚本

当用例集稳定后（通常第 2–3 次迭代之后），把"打分"这一步从人工换成脚本。这是知识工程从手工活变成工程化的分水岭。

读取 eval-set.json → 调用 LLM (current KB) → 按 rubric 自动打分 → 输出 v1 vs v2 对比 csv

输出建议包含三列：case_id / v_n_score / fail_reason。fail_reason 是金子——它直接告诉你下一轮该补哪条知识。

2 定义「评分规则」（Rubric） Scoring

核心一句话：每道题都拆成"规范分 + 业务分"两栏，不能合成一个总分。否则版本对比时你说不清是规范涨了还是业务理解涨了——两条曲线必须各自独立。

规范分 满分 4–6
  • 组件 / Variable / Text Style 是否绑定
  • 图标是否命中对照表
  • 间距是否走 spacing token
  • L1 / L3 题的主战场
业务分 满分 4–6
  • 主操作 / 信息优先级是否正确
  • 异常态、边界态是否被覆盖
  • 该出现的提示是否提前出现
  • L2 / L4 题的主战场
i

每个 Agent 的"业务分"细则要自己写一份；规范分细则三个 Agent 共用一份。具体打分点在 Step 1.3 范例库的 L2-A1 / L2-A2 / L2-A3 三道题里有示范。

3 建立「基线」并跑出对比 Baseline & Re-eval

用当前的知识库把全部用例跑一遍，逐条打分，得到 v1 基线；改完后用完全相同的题再跑一遍，得到 v2。

v1 → v2 评估对比示例（三 Agent 各取一题）

用例 v1 规范 / 业务 v2 规范 / 业务 Δ 主要修复点
出行向导 · 拥堵绕行 3 / 2 4 / 5 +4 把"绕 / 不绕"提为主按钮，时间代价加重
POI · 咖啡馆详情 4 / 3 4 / 5 +2 分区清晰化，闭店态加状态标记
点单餐饮 · 4 屏流程 2 / 2 4 / 5 +5 组件复用统一，打烊提示前置到加购
均分 3 / 2.3 4 / 5 +3.7 —
↑

规范分 3 → 4，业务分 2.3 → 5。分两条线看一眼就懂："规范基本封顶，业务理解才是这一版的真正进步。"——这种话才是可量化、可解释的训练成果。

4 针对性改进 —— 这才是"训练" Improve

哪条线最低，就集中改哪条线。改的不是模型，而是知识库与规则。

发现的短板 改进动作 改的是什么
三个 agent 都把决策动作做成了次按钮（业务分低） 在每个 agent 的业务规范里写死："唯一决策动作必须是主按钮，且不可折叠" Agent 业务知识库
POI 闭店、餐饮打烊等异常态 AI 经常忘做（L4 全跪） 在每个 agent 文档里加"必须覆盖的异常态清单"，并把它做成 always-applied rule Agent 业务知识库 + 规则权重
跨 agent 的按钮 / 图标语义不一致（L3 低） 抽出一份"三 agent 共用按钮 / 图标语义表"，作为公共卷的强制依据 设计系统主知识库
间距 / token 偶尔遗漏（规范分小幅扣） SOP 增加"自绘节点交付前 100% 绑 token"的强制自查项 设计系统主知识库

改完后回到第 3 步，用同一套题重新跑分——这就是闭环。

"装模作样" vs. "真正训练"

装模作样

  • 没有固定测试集，凭感觉改
  • 改完不知道有没有变好
  • 说不清当前水平、说不清下一步

真正在训练

  • 有固定题库，按 L1–L4 × 颗粒度组织
  • 每题"规范分 + 业务分"双线打分
  • 能说出"规范 3→4、业务 2.3→5"

结语

从今天起，把"和 AI 对话来优化知识库"变成有目标、有度量、有版本号的迭代—— 题库 + 评分 + 版本对比，三件事就足够撑起这套循环。

模型本身是冻住的黑盒，你优化的是黑盒外面的一切——这是一份纯粹的工程能力建设工作，价值并不比微调低。
